# Terraform

## 1장 IaC와 테라폼

### 1.1 인프라 자동화의 성숙도 변화
* **전통적인 인프라 구축 (On-Premise):** 과거에는 엔지니어가 물리 서버를 직접 조립하고 운영체제(OS) 및 네트워크를 수동으로 구성하여 많은 시간과 비용이 소모되었습니다.
* **클라우드의 등장:** AWS 등 클라우드 컴퓨팅이 대중화되면서 웹 콘솔 클릭 몇 번만으로 빠르게 인프라를 생성할 수 있게 되었습니다.
* **대규모 인프라의 한계:** 관리해야 할 서버가 수백 대 이상으로 늘어나면 웹 콘솔을 통한 수동 제어는 실수를 유발하고 효율성이 급격히 떨어집니다.
* **자동화와 IaC의 탄생:** 초기에는 셸 스크립트 기반의 자동화가 도입되었으나 이 역시 관리가 까다로웠습니다. 이를 극복하기 위해 인프라 자체를 하나의 코드로 정의하여 관리하는 **IaC (Infrastructure as Code, 코드로서의 인프라)** 기술이 등장했습니다.

---

### 1.2 프로세스로서의 자동화
* 자동화는 단순히 단발성 스크립트를 작성하여 실행하는 행위에 그치지 않습니다. 
* 인프라의 기획, 생성, 변경, 삭제에 이르는 전체 라이프사이클을 체계적으로 제어하는 **프로세스 자동화** 관점으로 접근해야 합니다.

---

### 1.3 IaC의 이해
* **정의:** 가상화된 인프라, 네트워크, 운영체제 등을 사람이 읽을 수 있는 프로그래밍 코드 형태로 정의하고 생성하는 기술입니다. AWS 콘솔에서 일일이 생성하던 VPC, 서브넷, EC2 등의 자원을 코드 한 줄(또는 몇 줄의 선언문)로 구성할 수 있습니다.
* **IaC 도입의 핵심 장점:**
  1. **동일한 환경 생성:** 개발(Dev), 테스트(Stage), 운영(Prod) 환경을 완벽히 동일하게 복제하여 환경 불일치로 인한 오류를 방지합니다.
  2. **버전 관리 및 추적:** 인프라 코드를 Git과 같은 버전 관리 시스템에 저장하므로, 누가 언제 어떤 자원을 수정했는지 명확하게 파악할 수 있습니다.
  3. **안정적인 재현 가능성:** 장애나 실수로 인해 서버가 손실되더라도, 작성해 둔 코드를 다시 실행하는 것만으로 신속한 원상 복구가 가능합니다.

---

### 1.4 테라폼(Terraform)의 특성
테라폼은 글로벌 보안/인프라 기업인 하시코프(HashiCorp)에서 개발한 가장 대표적인 오픈소스 IaC 도구입니다.

* **멀티 클라우드 지원:** AWS, Azure, GCP 등 서로 다른 퍼블릭 클라우드뿐만 아니라 다양한 플랫폼의 인프라를 테라폼이라는 하나의 도구와 언어(HCL)로 통합 관리할 수 있습니다.
* **선언형 방식 (Declarative):** 인프라를 생성하기 위한 세부 단계를 일일이 프로그래밍하는 방식이 아닙니다. 최종적으로 '원하는 인프라의 상태'를 코드로 선언하면, 테라폼이 현재 상태를 분석하여 목적지에 도달하기 위한 최적의 과정을 알아서 처리합니다.
* **State(상태 파일) 관리:** 테라폼의 핵심 메커니즘입니다. 실제 배포된 인프라의 현재 상태와 개발자가 코드로 작성한 원하는 상태를 1:1로 비교 및 기록하는 `.tfstate` 파일을 통해 인프라의 정밀한 동기화를 유지합니다.

---

### 1.5 테라폼 제공 유형
* **오픈소스 테라폼 (Terraform CLI):** 사용자의 로컬 환경이나 CI/CD 파이프라인에서 무료로 다운로드하여 사용하는 터미널 기반의 기본 도구입니다.
* **HCP Terraform (구 Terraform Cloud):** 하시코프가 직접 제공하는 SaaS(형상 관리 및 협업 인프라) 플랫폼으로, 웹 UI, 상태 파일 원격 저장, 팀 커뮤니팅 기능 등을 종합적으로 제공합니다.

---

### 1.6 테라폼과 다른 자동화 도구 비교
인프라 관리 도구는 역할에 따라 크게 두 가지 영역으로 나뉘며, 실무에서는 두 도구를 상호 보완적으로 함께 연동하여 사용합니다.

* **Ansible (구성 관리 - Configuration Management):** 주요 목적은 서버 내부 설정입니다. 이미 생성되어 있는 서버에 접속하여 필요한 패키지를 설치하고, 파일을 배포하며, OS 환경 설정을 변경하는 데 특화되어 있습니다.
* **Terraform (인프라 프로비저닝 - Orchestration):** 주요 목적은 인프라 자원의 생성입니다. 아무것도 없는 상태에서 VPC 네트워크 파이프라인을 뚫고, EC2 서버 인스턴스를 올리며, RDS 데이터베이스나 ALB 부하분산기 등의 아키텍처 뼈대를 구축하는 데 특화되어 있습니다.

---

### 1.7 테라폼 사용 목적과 실무 과제

#### 1. 주요 사용 목적
* **표준화:** 사내 모든 프로젝트의 인프라 아키텍처를 동일한 규격으로 표준화합니다.
* **자동화:** 반복적인 인프라 구성 작업을 제거하여 엔지니어의 공수를 줄이고 배포 속도를 높입니다.
* **협업 강화:** 동료들과 Git을 통해 코드를 공유하고 풀 리퀘스트(PR) 검토 과정을 거치며 안전하게 인프라를 발전시킵니다.
* **재사용성 향상:** 자주 사용하는 인프라 패턴을 모듈(Module)화하여 필요할 때마다 레고 블록처럼 쉽게 재사용합니다.

#### 2. 실무 도입 시 해결해야 할 과제
* **State 관리의 위험성:** 현재 인프라 상태를 기록한 State 파일이 유실되거나 동시 수정으로 인해 꼬이게 되면 운영 중인 서비스에 치명적인 문제가 생길 수 있으므로 원격 잠금(Locking) 관리가 필수적입니다.
* **코드 품질 유지:** IaC 역시 엄연한 프로그래밍 '코드'이므로 가독성 있는 작성 규칙을 준수해야 하며, 인프라 반영 전 팀원 간의 코드 리뷰 프로세스가 안착되어야 합니다.
* **보안 및 기밀 정보 관리:** 인프라 코드 내부에 데이터베이스 비밀번호나 클라우드 접근을 위한 Access Key 등의 민감 정보가 하드코딩되어 외부 Git 저장소에 노출되지 않도록 별도의 환경 변수나 보안 관리 도구를 연동해야 합니다.

## 2장 실행 환경 구성

### 2.1 테라폼 환경 구성
테라폼을 사용하려면 각 운영체제(OS)에 맞는 프로그램을 설치하고 실행 환경을 구성해야 합니다.

#### 2.1.1 테라폼 설치
* **설치 확인:** 터미널이나 커맨드 창(CMD)에서 아래 명령어를 입력하여 테라폼이 정상적으로 설치되었는지 확인합니다.
  > terraform version

#### 2.1.2 리눅스와 macOS 사용자 환경 구성
* **설치 방법:** 리눅스 환경에서는 보통 wget, curl, apt, yum 등의 패키지 관리 도구를 사용하여 설치합니다.
* **실행 파일 위치 확인:** 테라폼 실행 파일이 시스템 내 어느 경로에 존재하는지 확인합니다.
  > which terraform
* **환경변수 PATH 확인:** 리눅스나 macOS는 터미널에 terraform 명령어를 입력했을 때 PATH에 등록된 위치를 순서대로 탐색합니다. 값이 올바르게 잡혀있는지 아래 명령어로 확인합니다.
  > echo $PATH

#### 2.1.3 윈도우 사용자 환경 구성
* **설치 및 확인:** 윈도우용 바이너리를 다운로드하여 환경변수(Path)를 설정한 후, 명령 프롬프트(CMD)를 실행하여 정상 설치 여부를 확인합니다.
  > terraform version

---

### 2.2 IDE 구성
* **IDE (통합 개발 환경):** 코드를 효율적으로 작성하고 편집하기 위한 프로그램입니다.
* **대표적인 도구:** 테라폼 개발 시에는 가장 대중적이고 생태계가 넓은 **VS Code (Visual Studio Code)** 가 대표적으로 사용됩니다. (HashiCorp 공식 테라폼 확장 기능을 설치하면 자동 완성 및 구문 하이라이팅을 지원받을 수 있습니다.)

---

### 2.3 CLI 구성 파일
* **개요:** 테라폼 CLI의 동작 방식을 사용자 정의할 수 있는 전역 설정 파일입니다.
* **역할:** 주로 HCP Terraform(구 Terraform Cloud)과의 연동을 위한 자격 증명(Token) 저장, 제공자(Provider) 플러그인의 로컬 캐시 경로 지정 등 CLI 자체의 환경을 제어할 때 사용됩니다.
* **파일 위치:**
  * *Windows:* %APPDATA%\terraform.rc
  * *Linux / macOS:* ~/.terraformrc

## 3장 기본 사용법

### 3.1 주요 커맨드
테라폼을 사용할 때 가장 기본적이고 핵심이 되는 6가지 명령어는 `init`, `validate`, `plan`, `apply`, `destroy`, `fmt` 입니다.

#### 테라폼 핵심 작업 흐름 (Workflow)
1. **코드 작성** (인프라 정의)
2. **init** (프로젝트 초기화)
3. **validate** (코드 문법 검사)
4. **plan** (변경 사항 사전 실행 및 예측)
5. **apply** (실제 인프라 생성 및 반영)
6. **운영** (인프라 관리)
7. **destroy** (필요 시 인프라 전체 삭제)

---

#### 3.1.1 커맨드 사용법과 help 옵션
* 테라폼의 전체 명령어 목록이나 특정 명령어의 상세한 사용법(옵션 등)을 확인하고 싶을 때는 도움말 옵션을 사용합니다.
  > terraform -help
  > terraform plan -help

#### 3.1.2 init (초기화)
* **개요:** 테라폼 프로젝트를 시작할 때 가장 먼저, 그리고 반드시 실행해야 하는 명령어입니다.
* **주요 역할:** * 테라폼 코드가 클라우드(예: AWS)와 통신할 수 있도록 해주는 플러그인인 **프로바이더(Provider)**를 다운로드합니다.
  * 정의된 모듈(Module)을 다운로드하고 작업 환경을 초기화합니다.
* **주의사항:** 새로운 테라폼 프로젝트를 복제(Clone)받았거나 원격 저장소에서 새로 가져왔다면 무조건 `terraform init`부터 실행해야 합니다.
  > terraform init

#### 3.1.3 validate (검사)
* **개요:** 작성한 테라폼 코드의 서식이나 문법적인 오류가 없는지 검사합니다.
* **역할:** 실제 인프라에 반영하기 전, 코드 자체의 결함을 빠르게 잡아낼 수 있습니다.
  > terraform validate

#### 3.1.4 plan & apply (계획 및 반영)
* **plan (예측):** 테라폼이 실제 인프라를 변경하기 전에 어떤 자원이 생성, 수정, 삭제되는지 미리 보여주는 '시뮬레이션' 단계입니다. 운영 환경의 안전을 위해 필수적으로 확인해야 합니다.
  > terraform plan
* **apply (반영):** `plan`으로 확인한 변경 사항을 실제 클라우드 인프라에 적용하여 자원을 생성하거나 수정합니다.
  > terraform apply

#### 3.1.5 destroy (삭제)
* **개요:** 테라폼으로 생성하고 관리하던 모든 인프라 자원을 한 번에 안전하게 삭제합니다.
* **주의사항:** 실행 시 실제 운영 중인 모든 자원이 파괴되므로 극도로 주의해서 사용해야 합니다.
  > terraform destroy

#### 3.1.6 fmt (정렬)
* **개요:** 테라폼 코드의 띄어쓰기, 가독성, 정렬 상태를 HashiCorp 표준 스타일에 맞춰 자동으로 깔끔하게 정렬해 줍니다.
* **역할:** 협업 시 개발자 간의 코드 스타일을 일관되게 유지해 줍니다.
  > terraform fmt

### 3.2 HCL (HashiCorp Configuration Language)
테라폼을 배우면서 가장 먼저 익숙해져야 하는 언어가 바로 **HCL**입니다. 쉽게 말해 **테라폼 전용 인프라 설정 언어**입니다.

* **HCL이란?** 예를 들어 AWS 콘솔(웹 UI)에 접속하여 클릭으로 EC2 인스턴스의 이름, AMI, 인스턴스 타입을 선택하고 생성하는 모든 과정을 테라폼에서는 하나의 텍스트 코드로 기술합니다. 이때 사용하는 코드가 바로 HCL입니다.

#### 3.2.1 HCL을 사용하는 이유
1. **높은 가독성:** 사람이 읽고 이해하기 매우 쉽고 직관적입니다.
2. **인프라 특화:** 클라우드 자원을 정의하고 선언하는 데 최적화되어 있습니다.
3. **유연성과 재사용성:** 단순한 설정 파일 역할을 넘어, 변수나 반복문 등 프로그래밍 기능이 포함되어 있어 코드를 재사용하기 좋습니다.

---

#### HCL의 기본 구조
> resource "aws_instance" "web" {
>   ami           = "ami-123456"
>   instance_type = "t3.micro"
> }

* **resource:** 블록 유형 (무엇을 만들 것인가? 정의)
* **"aws_instance":** 리소스 유형 (생성할 클라우드 자원의 종류)
* **"web":** 테라폼 내부에서 이 리소스를 식별하기 위한 임의의 이름 (리소스 식별자)
* **ami = "ami-123456":** 자원에 할당할 구체적인 설정값 (Arguments)

---

#### 3.2.2 HCL 표현식 (Expressions)
표현식은 값을 직접 입력하거나, 계산 및 다른 자원을 참조할 때 사용하는 문법입니다.

* **문자열 (String):** 텍스트 데이터를 큰따옴표로 감싸서 표현합니다.
  > name = "web-server"
* **숫자 (Number):** 정수나 실수를 표현합니다.
  > port = 80
* **부울 (Boolean):** 참 또는 거짓을 나타냅니다.
  > enabled = true  # 또는 false
* **리스트 (List):** 대괄호 `[]` 안에 여러 값을 순서대로 저장합니다.
  > availability_zones = ["ap-northeast-2a", "ap-northeast-2c"]
* **맵 (Map):** 중괄호 `{}` 안에 키(Key)와 값(Value)의 쌍으로 저장하며, AWS 태그 설정을 할 때 자주 사용됩니다.
  > tags = { name = "web", env = "dev" }
* **참조 (Reference):** 다른 리소스의 속성값을 가져와 연결합니다.
  > vpc_id = aws_vpc.main.id  # 'main'이라는 VPC의 ID 값을 자동으로 참조
* **연산자 (Operators):** 사칙연산 및 비교 연산을 수행합니다.
* **조건식 (Conditional):** 삼항 연산자 형태로 조건에 따라 다른 값을 선택합니다.
  > instance_type = var.env == "prod" ? "t3.large" : "t3.micro"
* **함수 (Functions):** 테라폼 내장 함수를 이용해 데이터를 변환하거나 계산합니다.
  > length(["a", "b", "c"])  # 결과값: 3
