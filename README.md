# Terraform

## 1장 IaC와 테라폼

### 1.1 인프라 자동화의 성숙도 변화
* **전통적인 인프라 구축 (On-Premise):** 과거에는 엔지니어가 물리 서버를 직접 조립하고 운영체제(OS) 및 네트워크를 수동으로 구성하여 많은 시간과 비용이 소모되었습니다.
* **클라우드의 등장:** AWS 등 클라우드 컴퓨팅이 대중화되면서 웹 콘솔 클릭 몇 번만으로 빠르게 인프라를 생성할 수 있게 되었습니다.
* **대규모 인프라의 한계:** 관리해야 할 서버가 수백 대 이상으로 늘어나면 웹 콘솔을 통한 수동 제어는 실수를 유발하고 효율성이 급격히 떨어집니다.
* **자동화와 IaC의 탄생:** 초기에는 셸 스크립트 기반의 자동화가 도입되었으나 이 역시 관리가 까다로웠습니다. 이를 극복하기 위해 인프라 자체를 하나의 코드로 정의하여 관리하는 **IaC (Infrastructure as Code, 코드로서의 인프라)** 기술이 등장했습니다.

---

### 1.2 프로세스로서의 자동화
* 자동화는 단순히 단발성 스크립트를 작성하여 실행하는 행위에 그치지 않습니다. 
* 인프라의 기획, 생성, 변경, 삭제에 이르는 전체 라이프사이클을 체계적으로 제어하는 **프로세스 자동화** 관점으로 접근해야 합니다.

---

### 1.3 IaC의 이해
* **정의:** 가상화된 인프라, 네트워크, 운영체제 등을 사람이 읽을 수 있는 프로그래밍 코드 형태로 정의하고 생성하는 기술입니다. AWS 콘솔에서 일일이 생성하던 VPC, 서브넷, EC2 등의 자원을 코드 한 줄(또는 몇 줄의 선언문)로 구성할 수 있습니다.
* **IaC 도입의 핵심 장점:**
  1. **동일한 환경 생성:** 개발(Dev), 테스트(Stage), 운영(Prod) 환경을 완벽히 동일하게 복제하여 환경 불일치로 인한 오류를 방지합니다.
  2. **버전 관리 및 추적:** 인프라 코드를 Git과 같은 버전 관리 시스템에 저장하므로, 누가 언제 어떤 자원을 수정했는지 명확하게 파악할 수 있습니다.
  3. **안정적인 재현 가능성:** 장애나 실수로 인해 서버가 손실되더라도, 작성해 둔 코드를 다시 실행하는 것만으로 신속한 원상 복구가 가능합니다.

---

### 1.4 테라폼(Terraform)의 특성
테라폼은 글로벌 보안/인프라 기업인 하시코프(HashiCorp)에서 개발한 가장 대표적인 오픈소스 IaC 도구입니다.

* **멀티 클라우드 지원:** AWS, Azure, GCP 등 서로 다른 퍼블릭 클라우드뿐만 아니라 다양한 플랫폼의 인프라를 테라폼이라는 하나의 도구와 언어(HCL)로 통합 관리할 수 있습니다.
* **선언형 방식 (Declarative):** 인프라를 생성하기 위한 세부 단계를 일일이 프로그래밍하는 방식이 아닙니다. 최종적으로 '원하는 인프라의 상태'를 코드로 선언하면, 테라폼이 현재 상태를 분석하여 목적지에 도달하기 위한 최적의 과정을 알아서 처리합니다.
* **State(상태 파일) 관리:** 테라폼의 핵심 메커니즘입니다. 실제 배포된 인프라의 현재 상태와 개발자가 코드로 작성한 원하는 상태를 1:1로 비교 및 기록하는 `.tfstate` 파일을 통해 인프라의 정밀한 동기화를 유지합니다.

---

### 1.5 테라폼 제공 유형
* **오픈소스 테라폼 (Terraform CLI):** 사용자의 로컬 환경이나 CI/CD 파이프라인에서 무료로 다운로드하여 사용하는 터미널 기반의 기본 도구입니다.
* **HCP Terraform (구 Terraform Cloud):** 하시코프가 직접 제공하는 SaaS(형상 관리 및 협업 인프라) 플랫폼으로, 웹 UI, 상태 파일 원격 저장, 팀 커뮤니팅 기능 등을 종합적으로 제공합니다.

---

### 1.6 테라폼과 다른 자동화 도구 비교
인프라 관리 도구는 역할에 따라 크게 두 가지 영역으로 나뉘며, 실무에서는 두 도구를 상호 보완적으로 함께 연동하여 사용합니다.

* **Ansible (구성 관리 - Configuration Management):** 주요 목적은 서버 내부 설정입니다. 이미 생성되어 있는 서버에 접속하여 필요한 패키지를 설치하고, 파일을 배포하며, OS 환경 설정을 변경하는 데 특화되어 있습니다.
* **Terraform (인프라 프로비저닝 - Orchestration):** 주요 목적은 인프라 자원의 생성입니다. 아무것도 없는 상태에서 VPC 네트워크 파이프라인을 뚫고, EC2 서버 인스턴스를 올리며, RDS 데이터베이스나 ALB 부하분산기 등의 아키텍처 뼈대를 구축하는 데 특화되어 있습니다.

---

### 1.7 테라폼 사용 목적과 실무 과제

#### 1. 주요 사용 목적
* **표준화:** 사내 모든 프로젝트의 인프라 아키텍처를 동일한 규격으로 표준화합니다.
* **자동화:** 반복적인 인프라 구성 작업을 제거하여 엔지니어의 공수를 줄이고 배포 속도를 높입니다.
* **협업 강화:** 동료들과 Git을 통해 코드를 공유하고 풀 리퀘스트(PR) 검토 과정을 거치며 안전하게 인프라를 발전시킵니다.
* **재사용성 향상:** 자주 사용하는 인프라 패턴을 모듈(Module)화하여 필요할 때마다 레고 블록처럼 쉽게 재사용합니다.

#### 2. 실무 도입 시 해결해야 할 과제
* **State 관리의 위험성:** 현재 인프라 상태를 기록한 State 파일이 유실되거나 동시 수정으로 인해 꼬이게 되면 운영 중인 서비스에 치명적인 문제가 생길 수 있으므로 원격 잠금(Locking) 관리가 필수적입니다.
* **코드 품질 유지:** IaC 역시 엄연한 프로그래밍 '코드'이므로 가독성 있는 작성 규칙을 준수해야 하며, 인프라 반영 전 팀원 간의 코드 리뷰 프로세스가 안착되어야 합니다.
* **보안 및 기밀 정보 관리:** 인프라 코드 내부에 데이터베이스 비밀번호나 클라우드 접근을 위한 Access Key 등의 민감 정보가 하드코딩되어 외부 Git 저장소에 노출되지 않도록 별도의 환경 변수나 보안 관리 도구를 연동해야 합니다.

## 2장 실행 환경 구성

### 2.1 테라폼 환경 구성
테라폼을 사용하려면 각 운영체제(OS)에 맞는 프로그램을 설치하고 실행 환경을 구성해야 합니다.

#### 2.1.1 테라폼 설치
* **설치 확인:** 터미널이나 커맨드 창(CMD)에서 아래 명령어를 입력하여 테라폼이 정상적으로 설치되었는지 확인합니다.
  > terraform version

#### 2.1.2 리눅스와 macOS 사용자 환경 구성
* **설치 방법:** 리눅스 환경에서는 보통 wget, curl, apt, yum 등의 패키지 관리 도구를 사용하여 설치합니다.
* **실행 파일 위치 확인:** 테라폼 실행 파일이 시스템 내 어느 경로에 존재하는지 확인합니다.
  > which terraform
* **환경변수 PATH 확인:** 리눅스나 macOS는 터미널에 terraform 명령어를 입력했을 때 PATH에 등록된 위치를 순서대로 탐색합니다. 값이 올바르게 잡혀있는지 아래 명령어로 확인합니다.
  > echo $PATH

#### 2.1.3 윈도우 사용자 환경 구성
* **설치 및 확인:** 윈도우용 바이너리를 다운로드하여 환경변수(Path)를 설정한 후, 명령 프롬프트(CMD)를 실행하여 정상 설치 여부를 확인합니다.
  > terraform version

---

### 2.2 IDE 구성
* **IDE (통합 개발 환경):** 코드를 효율적으로 작성하고 편집하기 위한 프로그램입니다.
* **대표적인 도구:** 테라폼 개발 시에는 가장 대중적이고 생태계가 넓은 **VS Code (Visual Studio Code)** 가 대표적으로 사용됩니다. (HashiCorp 공식 테라폼 확장 기능을 설치하면 자동 완성 및 구문 하이라이팅을 지원받을 수 있습니다.)

---

### 2.3 CLI 구성 파일
* **개요:** 테라폼 CLI의 동작 방식을 사용자 정의할 수 있는 전역 설정 파일입니다.
* **역할:** 주로 HCP Terraform(구 Terraform Cloud)과의 연동을 위한 자격 증명(Token) 저장, 제공자(Provider) 플러그인의 로컬 캐시 경로 지정 등 CLI 자체의 환경을 제어할 때 사용됩니다.
* **파일 위치:**
  * *Windows:* %APPDATA%\terraform.rc
  * *Linux / macOS:* ~/.terraformrc

## 3장 기본 사용법

### 3.1 주요 커맨드
테라폼을 사용할 때 가장 기본적이고 핵심이 되는 6가지 명령어는 `init`, `validate`, `plan`, `apply`, `destroy`, `fmt` 입니다.

#### 테라폼 핵심 작업 흐름 (Workflow)
1. **코드 작성** (인프라 정의)
2. **init** (프로젝트 초기화)
3. **validate** (코드 문법 검사)
4. **plan** (변경 사항 사전 실행 및 예측)
5. **apply** (실제 인프라 생성 및 반영)
6. **운영** (인프라 관리)
7. **destroy** (필요 시 인프라 전체 삭제)

---

#### 3.1.1 커맨드 사용법과 help 옵션
* 테라폼의 전체 명령어 목록이나 특정 명령어의 상세한 사용법(옵션 등)을 확인하고 싶을 때는 도움말 옵션을 사용합니다.
  > terraform -help
  > terraform plan -help

#### 3.1.2 init (초기화)
* **개요:** 테라폼 프로젝트를 시작할 때 가장 먼저, 그리고 반드시 실행해야 하는 명령어입니다.
* **주요 역할:** * 테라폼 코드가 클라우드(예: AWS)와 통신할 수 있도록 해주는 플러그인인 **프로바이더(Provider)**를 다운로드합니다.
  * 정의된 모듈(Module)을 다운로드하고 작업 환경을 초기화합니다.
* **주의사항:** 새로운 테라폼 프로젝트를 복제(Clone)받았거나 원격 저장소에서 새로 가져왔다면 무조건 `terraform init`부터 실행해야 합니다.
  > terraform init

#### 3.1.3 validate (검사)
* **개요:** 작성한 테라폼 코드의 서식이나 문법적인 오류가 없는지 검사합니다.
* **역할:** 실제 인프라에 반영하기 전, 코드 자체의 결함을 빠르게 잡아낼 수 있습니다.
  > terraform validate

#### 3.1.4 plan & apply (계획 및 반영)
* **plan (예측):** 테라폼이 실제 인프라를 변경하기 전에 어떤 자원이 생성, 수정, 삭제되는지 미리 보여주는 '시뮬레이션' 단계입니다. 운영 환경의 안전을 위해 필수적으로 확인해야 합니다.
  > terraform plan
* **apply (반영):** `plan`으로 확인한 변경 사항을 실제 클라우드 인프라에 적용하여 자원을 생성하거나 수정합니다.
  > terraform apply

#### 3.1.5 destroy (삭제)
* **개요:** 테라폼으로 생성하고 관리하던 모든 인프라 자원을 한 번에 안전하게 삭제합니다.
* **주의사항:** 실행 시 실제 운영 중인 모든 자원이 파괴되므로 극도로 주의해서 사용해야 합니다.
  > terraform destroy

#### 3.1.6 fmt (정렬)
* **개요:** 테라폼 코드의 띄어쓰기, 가독성, 정렬 상태를 HashiCorp 표준 스타일에 맞춰 자동으로 깔끔하게 정렬해 줍니다.
* **역할:** 협업 시 개발자 간의 코드 스타일을 일관되게 유지해 줍니다.
  > terraform fmt

### 3.2 HCL (HashiCorp Configuration Language)
테라폼을 배우면서 가장 먼저 익숙해져야 하는 언어가 바로 **HCL**입니다. 쉽게 말해 **테라폼 전용 인프라 설정 언어**입니다.

* **HCL이란?** 예를 들어 AWS 콘솔(웹 UI)에 접속하여 클릭으로 EC2 인스턴스의 이름, AMI, 인스턴스 타입을 선택하고 생성하는 모든 과정을 테라폼에서는 하나의 텍스트 코드로 기술합니다. 이때 사용하는 코드가 바로 HCL입니다.

#### 3.2.1 HCL을 사용하는 이유
1. **높은 가독성:** 사람이 읽고 이해하기 매우 쉽고 직관적입니다.
2. **인프라 특화:** 클라우드 자원을 정의하고 선언하는 데 최적화되어 있습니다.
3. **유연성과 재사용성:** 단순한 설정 파일 역할을 넘어, 변수나 반복문 등 프로그래밍 기능이 포함되어 있어 코드를 재사용하기 좋습니다.

---

#### HCL의 기본 구조
> resource "aws_instance" "web" {
>   ami           = "ami-123456"
>   instance_type = "t3.micro"
> }

* **resource:** 블록 유형 (무엇을 만들 것인가? 정의)
* **"aws_instance":** 리소스 유형 (생성할 클라우드 자원의 종류)
* **"web":** 테라폼 내부에서 이 리소스를 식별하기 위한 임의의 이름 (리소스 식별자)
* **ami = "ami-123456":** 자원에 할당할 구체적인 설정값 (Arguments)

---

#### 3.2.2 HCL 표현식 (Expressions)
표현식은 값을 직접 입력하거나, 계산 및 다른 자원을 참조할 때 사용하는 문법입니다.

* **문자열 (String):** 텍스트 데이터를 큰따옴표로 감싸서 표현합니다.
  > name = "web-server"
* **숫자 (Number):** 정수나 실수를 표현합니다.
  > port = 80
* **부울 (Boolean):** 참 또는 거짓을 나타냅니다.
  > enabled = true  # 또는 false
* **리스트 (List):** 대괄호 `[]` 안에 여러 값을 순서대로 저장합니다.
  > availability_zones = ["ap-northeast-2a", "ap-northeast-2c"]
* **맵 (Map):** 중괄호 `{}` 안에 키(Key)와 값(Value)의 쌍으로 저장하며, AWS 태그 설정을 할 때 자주 사용됩니다.
  > tags = { name = "web", env = "dev" }
* **참조 (Reference):** 다른 리소스의 속성값을 가져와 연결합니다.
  > vpc_id = aws_vpc.main.id  # 'main'이라는 VPC의 ID 값을 자동으로 참조
* **연산자 (Operators):** 사칙연산 및 비교 연산을 수행합니다.
* **조건식 (Conditional):** 삼항 연산자 형태로 조건에 따라 다른 값을 선택합니다.
  > instance_type = var.env == "prod" ? "t3.large" : "t3.micro"
* **함수 (Functions):** 테라폼 내장 함수를 이용해 데이터를 변환하거나 계산합니다.
  > length(["a", "b", "c"])  # 결과값: 3

### 3.3 테라폼 블록
Terraform 코드에는 보통 `terraform { }` 블록이 포함되어 있으며, 이는 **테라폼 자체의 환경 설정**을 정의하는 역할을 합니다. 쉽게 말해 프로그램의 **'환경설정 파일'**과 같은 존재입니다.

ex) `terraform { required_version = ">= 1.0" }` 
> **의미:** 이 코드는 테라폼 버전 1.0 이상에서만 실행 가능하다는 것을 강제합니다.

---

#### 3.3.1 테라폼 버전
* **개요:** 협업 및 실무 환경에서 팀원 간 혹은 배포 시스템의 테라폼 버전을 고정하거나 제한합니다.
* **목적:** 예기치 못한 버전 업데이트로 인한 코드 오류나 오작동을 방지하기 위해 필수적으로 지정을 권장합니다.

#### 3.3.2 프로바이더 버전
* **개요:** 테라폼은 클라우드 서비스(AWS, GCP 등)와 직접 통신하지 않고, 중간에 **Provider**라는 매개체를 거쳐 API를 호출합니다.
  > `Terraform` ➔ `AWS Provider` ➔ `AWS API`
* **역할:** 실무에서는 프로바이더의 갑작스러운 업데이트로 기존 코드가 깨지는 것을 막기 위해 버전을 명시적으로 고정합니다.

* **설정 예시 (AWS 프로바이더 6.0 버전 사용):**

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 6.0"
    }
  }
}

* **`terraform init`과의 관계:**
  `terraform init` 명령어를 실행하면 테라폼은 이 `required_providers` 설정을 확인하고, 필요한 프로바이더 파일(예: `hashicorp/aws`)을 인터넷에서 자동으로 다운로드합니다.

#### 3.3.3 cloud 블록
* **개요:** **Terraform Cloud**(현재 명칭: *HCP Terraform*)와의 연동을 설정하는 블록입니다.
* **Terraform Cloud란?** 테라폼 전용 원격 협업 플랫폼으로, 인프라의 상태 파일(State) 저장, 팀원 간의 협업, 권한 관리, 자동 배포(CI/CD) 기능을 제공합니다.

* **설정 예시 (`my-org` 조직의 `prod` 작업공간 사용):**

terraform {
  cloud {
    organization = "my-org"
    workspaces {
      name = "prod"
    }
  }
}

#### 3.3.4 백엔드 블록
* **개요:** 테라폼이 생성한 실제 인프라 자원의 정보를 기억하는 저장소(**Backend**)를 지정합니다.
* **핵심 개념 이해하기:**
  * **State (`terraform.tfstate`):** 테라폼이 생성한 VPC, EC2, RDS 등의 정보를 담고 있는 일종의 **'기억장치(데이터)'**입니다. 지정하지 않으면 기본적으로 내 PC(로컬)에 저장됩니다.
  * **Backend:** 이 기억장치(State 파일)를 안전하게 보관하는 **'저장 장소'**입니다. 실무에서는 여러 개발자가 동일한 인프라를 안전하게 관리할 수 있도록 S3나 Terraform Cloud 같은 원격 저장소를 백엔드로 지정하여 공유합니다.
    

### 3.4 리소스 (Resource)
Terraform에서 **Resource**는 실제로 클라우드에 생성되는 물리적/논리적 인프라 자원을 의미합니다.
* **종류:** EC2, VPC, Subnet, Security Group, RDS, S3 등 마우스 클릭으로 생성하던 모든 것이 리소스입니다.
* **개념:** AWS 콘솔에서 화면을 클릭하며 만들던 인프라를 테라폼에서는 코드를 통해 정의하고 생성합니다.

---

#### 3.4.1 리소스 구성

**1) 기본 구조**
resource "리소스타입" "이름" {
  # 설정 속성 (Arguments)
}

**2) 작성 예시**
resource "aws_instance" "web" {
  ami           = "ami-123456"
  instance_type = "t3.micro"
}

**3) 구조 분석**
* **`resource`:** 테라폼 리소스를 선언하는 키워드입니다.
* **`aws_instance`:** 리소스의 종류(Type)를 뜻하며, 여기서는 AWS의 EC2 인스턴스를 의미합니다.
* **`web`:** 테라폼 코드 내부에서 이 리소스를 식별하기 위해 사용자가 임의로 지정하는 이름입니다.

---

**4) 주요 AWS 리소스 예시**

* **VPC 생성**
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

* **Subnet 생성**
resource "aws_subnet" "public" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "10.0.1.0/24"
}

* **EC2 인스턴스 생성**
resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t3.micro"
}

> **참고:** 실무 환경에서는 이러한 수십, 수백 개의 리소스들을 유기적으로 연결하여 관리하게 됩니다.

---

#### 3.4.2 종속성 (Dependency)
인프라 구축에는 순서가 필요합니다. (예: `VPC 생성` ➔ `Subnet 생성` ➔ `EC2 생성`)
* **암묵적 종속성 (Implicit Dependency):** 위 Subnet 예시처럼 `vpc_id = aws_vpc.main.id` 형태로 코드가 다른 리소스를 참조하면, 테라폼이 이를 자동으로 파악합니다. "VPC가 먼저 있어야 Subnet을 만들 수 있구나"라고 판단하여 순서대로 인프라를 생성합니다.

#### 3.4.3 리소스 속성 참조
리소스를 생성하고 나면 테라폼은 해당 자원의 세부 정보(VPC ID, ARN, CIDR 등)를 자동으로 추적하고 기억합니다. 덕분에 다른 리소스를 생성할 때 해당 결괏값(ID 등)을 간편하게 가져와 연결할 수 있습니다.

#### 3.4.4 수명주기 (Lifecycle)
인프라 자원이 생성, 변경, 삭제될 때의 상세 동작을 제어하는 옵션입니다.

* **설정 예시:**
resource "aws_instance" "web" {
  # ... 인스턴스 설정 생략 ...

  lifecycle {
    create_before_destroy = true
  }
}

* **핵심 옵션 설명:**
  * **`create_before_destroy = true`:** 테라폼의 기본 동작은 기존 자원을 '삭제한 후 새 자원을 생성'하는 것입니다. 이 옵션을 켜면 **새 자원을 먼저 생성한 뒤 기존 자원을 삭제**하므로, 인프라 교체 시 발생하는 서비스 중단(Downtime)을 방지할 수 있습니다.
  * **`prevent_destroy = true`:** 실수로 리소스가 삭제되는 것을 절대적으로 금지합니다. (데이터베이스 등 중요 자원에 필수)
  * **`ignore_changes`:** 외부 요인이나 특정 속성의 변경 사항을 테라폼이 감지하지 않고 무시하도록 설정합니다.

### 3.5 데이터 소스 (Data Source)
데이터 소스는 인프라를 새로 생성하지 않고, **이미 클라우드에 존재하는 정보를 조회하여 가져오는 기능**입니다.
* **개념:** AWS 콘솔에 이미 만들어져 있는 VPC, AMI, Subnet, Security Group 등을 테라폼 코드로 읽어와서 사용하고 싶을 때 활용합니다.

> **💡 Resource vs Data Source 핵심 차이**
> * **Resource:** 인프라를 **새로 생성**할 때 사용 (예: 새로운 VPC 만들기)
> * **Data Source:** 이미 생성되어 있는 인프라를 **조회**할 때 사용 (예: 기존 VPC ID 알아오기)

---

#### 3.5.1 데이터 소스 구성

**1) 기본 구조**
data "종류" "이름" {
  # 조건을 지정하여 원하는 자원을 필터링합니다.
}

**2) 작성 예시 (최신 우분투 AMI 조회)**
data "aws_ami" "ubuntu" {
  most_recent = true

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-focal-20.04-amd64-server-*"]
  }
}

---

#### 3.5.2 데이터 소스 속성 참조
Resource 블록처럼 Data Source 블록도 조회 결과에 따른 고유한 속성(ID, ARN 등)을 가집니다. 다른 리소스에서 이 값을 가져와 연결할 수 있습니다.

* **속성 참조 문법:** `data.종류.이름.속성` (예: `data.aws_vpc.main.id`)

**🔍 Resource 참조 vs Data Source 참조 비교**
* **Resource 참조:** `aws_vpc.main.id` (내가 코드로 만든 VPC의 ID)
* **Data Source 참조:** `data.aws_vpc.main.id` (이미 있던 VPC를 조회해서 가져온 ID)

---

#### 3.5.3 실무에서 자주 사용하는 데이터 소스
실무 환경에서는 인프라를 유기적으로 결합하기 위해 다음과 같은 데이터 소스를 매우 자주 사용합니다.

* **`aws_ami`:** 인스턴스 생성에 필요한 최신 OS 이미지(AMI) 정보 조회
* **`aws_vpc`:** 이미 구축되어 있는 기존 VPC 정보 조회
* **`aws_subnet`:** 특정 서브넷 정보 및 ID 조회
* **`aws_security_group`:** 기존에 정의된 보안 그룹 정보 조회
* **`aws_caller_identity`:** 현재 테라폼을 실행 중인 AWS 계정 ID(Account ID) 및 자격증명 정보 조회

### 3.6 입력 변수 (Variable)
* **필요성:** 테라폼에서 변수는 **'설정값의 분리'**를 위해 사용합니다. 인프라 설정 코드와 실제 입력값을 분리함으로써 하나의 코드로 개발(Dev), 운영(Prod) 등 다양한 환경에 재사용할 수 있습니다.

---

#### 3.6.1 변수 선언 방식

**1) 기본 구조**
variable "변수_이름" {
  type        = 자료형
  default     = 기본값
  description = "변수에 대한 설명"
}

**2) 작성 예시**
variable "instance_type" {
  type        = string
  default     = "t3.micro"
  description = "EC2 인스턴스의 스펙을 정의합니다."
}

**3) 구조 분석**
* **`variable`:** 변수를 정의하겠다는 테라폼 키워드입니다.
* **`instance_type`:** 고유한 변수 이름입니다. 코드 내에서 이 이름을 통해 호출합니다.
* **`type`:** 변수에 들어올 데이터의 자료형을 강제합니다.
* **`default`:** 변수 값이 별도로 지정되지 않았을 때 기본으로 적용될 값입니다.

---

#### 3.6.2 변수 유형 (Data Types)
테라폼은 다음과 같은 다양한 타입의 변수를 지원합니다.

* **`string` (문자열):** `"t3.micro"`, `"ap-northeast-2"`와 같은 텍스트 데이터
* **`number` (숫자):** `3`, `80`과 같은 정수 및 실수 데이터
* **`bool` (불리언):** `true` 또는 `false` 값
* **`list` (리스트):** 순서가 있는 값들의 배열 (예: `["ap-northeast-2a", "ap-northeast-2c"]`)
* **`map` (맵):** 키-값(Key-Value) 쌍으로 구성된 데이터 (예: `{ dev = "t3.micro", prod = "m5.large" }`)

---

#### 3.6.3 유효성 검사 (Validation)
변수에 잘못된 값이 입력되어 인프라 생성이 실패하는 것을 방지하기 위해, 값이 들어오는 단계에서 미리 검사할 수 있습니다.

* **작성 예시:**
variable "environment" {
  type = string

  validation {
    condition     = contains(["dev", "stage", "prod"], var.environment)
    error_message = "환경명은 반드시 dev, stage, prod 중 하나여야 합니다."
  }
}

> **💡 실무 활용 사례:**
> * 허용된 특정 인스턴스 타입만 들어오도록 제한
> * CIDR 블록 형식이 올바른지 검사 (`10.0.0.0/16` 등)
> * 지정된 환경명(dev, prod) 검증

---

#### 3.6.4 변수 참조
정의한 변수를 리소스 블록 등에서 가져다 쓸 때는 **`var.변수명`** 형태로 참조합니다.

* **참조 예시:**
resource "aws_instance" "web" {
  ami           = "ami-123456"
  instance_type = var.instance_type  # 변수 참조 적용
}

---

#### 3.6.5 민감한 변수 취급 (Sensitive)
비밀번호, API 토큰, 데이터베이스 접속 정보와 같이 외부로 노출되면 안 되는 민감한 변수에는 `sensitive = true` 옵션을 부여합니다.

* **작성 예시:**
variable "db_password" {
  type      = string
  sensitive = true
}

* **효과:** 이 옵션이 적용된 변수는 `terraform plan`이나 `terraform apply` 실행 시 터미널 화면에 값이 노출되지 않고 **`(sensitive value)`**로 마스킹 처리되어 안전하게 보호됩니다.

---

#### 3.6.6 변수 입력 방식과 우선순위
테라폼은 변수 값을 외부에서 주입할 수 있는 다양한 경로를 제공하며, 여러 곳에서 동시에 입력될 경우 정해진 우선순위에 따라 최종 값이 결정됩니다.

**1) 주요 입력 방식**
* 실행 시점 입력: `terraform apply` 실행 후 터미널에 직접 타이핑하여 입력
* 명령줄 명령어 옵션: `terraform apply -var="instance_type=t3.medium"`
* 환경 변수 활용: 시스템 환경 변수에 `TF_VAR_instance_type` 등록
* 변수 정의 파일 활용: `terraform.tfvars` 파일에 `instance_type = "t3.medium"` 형태로 미리 저장 (**실무에서 가장 선호하는 방식**)

**2) 우선순위 (아래로 갈수록 우선순위가 높음)**
1. 환경 변수 (`TF_VAR_변수명`)
2. `terraform.tfvars` 파일
3. `*.auto.tfvars` 파일
4. CLI 명령줄 옵션 (`-var` 또는 `-var-file`)

### 3.7 로컬 변수 (Local)
로컬 변수는 **테라폼 코드 내부에서만 선언하고 사용하는 변수**입니다. 외부에서 값을 주입받는 `Variable`과 달리, 코드 안에서 값을 가공하거나 중복되는 자원 이름을 하나로 묶을 때 사용합니다.

---

#### 3.7.1 입력 변수(Variable) vs 로컬 변수(Local)

| 구분 | Variable (입력 변수) | Local (로컬 변수) |
| :--- | :---: | :---: |
| **외부 입력 가능 여부** | O | X |
| **사용자 값 변경 가능** | O | X |
| **코드 내부 계산/가공용** | △ | O |
| **실무 사용 빈도** | 매우 높음 | 매우 높음 |

* **Variable:** 사용자가 실행 시점에 입력하는 값 (ex: EC2 타입, 리전, 배포 환경 등)
* **Local:** 테라폼이 내부적으로 계산해서 사용하는 값 (ex: 공통 태그 이름, 규칙이 있는 리소스 이름, 문자열 조합 등)

---

#### 3.7.2 local 선언

**1) 기본 구조**
locals {
  변수_이름1 = 값1
  변수_이름2 = 값2
}

* **의미:** 테라폼 내부에서 쓸 변수들을 정의하며, 하나의 `locals` 블록 안에 여러 개의 변수를 동시에 선언할 수 있습니다.

**2) 실무 활용 예시 (문자열 조합 및 태그 정의)**
locals {
  team        = "devops"
  environment = "prod"
  
  # 변수와 일반 텍스트를 조합하여 내부 변수 생성
  instance_name = "${local.team}-${local.environment}-web"

  # 공통으로 사용할 태그 맵(Map) 정의
  common_tags = {
    ManagedBy = "Terraform"
    Owner     = local.team
  }
}

---

#### 3.7.3 local 참조
정의한 로컬 변수를 리소스에서 가져다 쓸 때는 **`local.변수이름`** 형태로 참조합니다. (※ 선언할 때는 `locals`이지만, 사용할 때는 단수형인 `local`을 사용합니다.)

* **참조 문법:** `local.이름`

* **참조 예시:**
resource "aws_instance" "web" {
  ami           = "ami-123456"
  instance_type = "t3.micro"

  tags = {
    Name = local.instance_name  # 'devops-prod-web' 이라는 이름이 주입됩니다.
  }
}
